Origin & database load — GCP Cloud Monitoring

The supply side tells the real capacity story. At the peak the API ran at its engineered ceiling — 20 of 20 Cloud Run instances and ~800 of 800 concurrent requests per instance — and the database saturated its (then) 100-connection cap. Notably, database CPU never broke 46%: the binding resource was connection & request slots, not throughput.

Why a viewer-count model understated the load by ~20×

In the clean post-incident wind-down, the API still held ~10 of 20 instances at 02:00 UTC when only ~276 viewers were active. The reason: every open tab holds a Server-Sent-Events connection (/v1/stream, ~249k connections in 24h) that occupies a server request slot for up to an hour — watching or not. Capacity is bound by cumulative open tabs, not instantaneous viewers.

Incident timeline 2026-06-01 UTC

Three viewer-facing incidents, all configuration/edge — not raw capacity — and all resolved same-day with no data loss.

15:00 — Pre-show ramp begins; traffic climbs from a standing start.
17:10–18:40 — Peak. API pinned at 20/20 instances; Cloud SQL connections climb to the 100 cap (stale-revision accumulation) → intermittent 5xx.
17:31–17:38 — Edge outage (7 min). The front-end Worker hits Cloudflare's free 100k-req/day cap (error 1027) — full site-shell blackout. Visible as a 7-minute hole in page views.
18:50–19:50 — 429 storm. Rate limiter keyed on Cloudflare's edge IP + Vary: Cookie defeating CDN coalescing → ~80% of reads rejected; 4xx peaks at ~99k / 10 min. Client retries (standings.retry) hit 951/hr.
20:00 — Resolved. DB tier bump to 200 connections (#173) + limiter/cache fix (#176) deploy. 429s, 5xx, and retries collapse to ~0.
20:10–21:25 — Zero-downtime schema re-key (#179/#181) and stats windowing (#185) land cleanly under live traffic — 0× 5xx through both rollovers.

Capacity & forward headroom

Huge headroom — not the constraint

Edge / Worker: 714k/day on a 10M/mo plan.
DB read throughput: ~15–23 req/s origin, CPU ≤46%, 86% edge-cached.
The pre-event model's headline fear — "DB read saturation" — never materialized.

The binding constraint — ~1.5–2× organic

SSE request slots (Cloud Run concurrency × instances) and DB connection slots (pool × instances × revisions). Both are long-lived-connection limits — they were at the ceiling at launch. Finals (potentially 2–5× launch) need the changes below.
